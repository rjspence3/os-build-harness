The home_banking clone is the harness's benchmark build: a known-good original (a real OutSystems
Home Banking app) that the runner regenerates recipe-by-recipe, with pixel/structure diffing against
the original to measure distance-from-spec. It is the canonical proof that the clone loop works.

## Loop doctrine (from seeded memory)

- **App-cloning loop mandate** — decompose app → recipes → Claude Code + Mentor loop → spec-complete;
  banking is the benchmark because the original is known-good to diff against. See memory
  `mcp_recipes_app_cloning_loop_mandate` and `mcp_recipes_loop_snapshot_and_beat0`.
- **Runtime-parity verification** — session-reuse auth (log into original, screenshot clone in the
  same session), identical viewport, mask extension popups, demo-picker for populated data;
  `pixel_diff` flags must be joined (e.g. `--tol=16`). See memory
  `clone_runtime_parity_verification_method` and `home_banking_demo_auth_mechanism`.

## Contents

- `MCP_RECIPES/` — the recipe library (00_app_shell … NN), per-app manifests under
  `apps/home_banking/`, and `_raw/` screen captures (`.tree.md`) the renderers + tests read.
- `theme/portal.css` — extracted Home Banking theme (absolute original resource URLs; pixel-identical).
- `banking_runner_out/` — rendered Mentor prompts (gitignored; regenerated by `build_banking.py`).
- `compare/` — clone-vs-original diff PNGs (gitignored; diagnostic).

## Run

```bash
# from repo root, with the venv active
python scripts/build_banking.py --app core --dry-run     # render prompts, no MCP calls
python scripts/build_banking.py --list-apps              # inspect manifests
```
